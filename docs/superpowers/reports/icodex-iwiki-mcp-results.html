1. Резюме спецификации

Спецификация делает stdio MCP-сервер iwiki-mcp встроенной, всегда включённой частью изолированного Codex-окружения, которое поднимает icodex. Секрет (IWIKI_LLM_KEY) держится вне git, остальные настройки фиксируются литерально в блоке сервера.

Требования по разделам

IDРазделТребование
R1ObjectiveКаждый Codex-home, который запускает icodex, имеет зарегистрированный MCP-сервер iwiki, всегда включённый; регистрация переживает ребилд.
R2Scope: TargetТолько Codex (icodex); iclaude/Claude-сторона не трогается.
R3Scope: Always onНет переключателя enable/disable — сервер прописывается на каждом запуске.
R4Scope: System binaryЖёстко прописанный путь /home/ikeniborn/.local/bin/iwiki-mcp; без изолированной установки и разрешения через PATH.
R5Scope: Settings fixedНесекретный env (IWIKI_BASE_DIR, IWIKI_LLM_BASE_URL, IWIKI_EMBED_MODEL, IWIKI_EMBED_DIMENSIONS) — литералом в блоке; вне блока только секрет.
R6Scope: No bindingБез per-project привязки (.iwiki.toml / IWIKI_PROJECT_DIR); блок полностью статичен.
R7Background / git-factsСекрет не может попасть в tracked-файлы; он живёт только в .codex_config и достаётся серверу через окружение (env_vars), без shell-раскрытия.
R8ArchitectureРешение повторяет существующую форму icodex: env-маппинг для секрета + per-launch wiring-шаг, поддерживающий ограниченный регион в per-home config.toml.
R9Component 1Статический ограниченный регион [mcp_servers.iwiki] с правильным порядком TOML (ключи до подтаблицы .env), добавляемый в конец файла.
R10Component 2.codex_config получает ICODEX_IWIKI_LLM_KEY=<secret>; .codex_config.example — закомментированный пример без значения.
R11Component 3lib/config/env.sh: разрешить ICODEX_IWIKI_* (raw IWIKI_* остаётся запрещён) и добавить apply_iwiki_env() (де-префикс в IWIKI_LLM_KEY, без перетирания).
R12Component 4Новый lib/iwiki/iwiki.sh с ensure_iwiki_wiring: идемпотентный resync региона (strip + append), без gate, с cmp -s-guard.
R13Component 5icodex.sh: подключить модуль, вызвать ensure_iwiki_wiring после ensure_idd_wiring и apply_iwiki_env после apply_api_key.
R14Error handlingОтсутствие секрета/бинарника не вызывает жёсткого падения на запуске; поведение задокументировано; запись региона безопасна (temp + compare + replace).
R15Testingtest_iwiki_wiring.sh и test_iwiki_env.sh покрывают обе новые функции (один регион, идемпотентность, замена stale, не-перетирание, no-op, allowlist).

Критерии успеха (Success Criteria)

SC2Повторный запуск — no-op (байт-идентичен); stale-регион заменяется, не дублируется; регион — завершающее содержимое файла.
SC3apply_iwiki_env экспортирует IWIKI_LLM_KEY из ICODEX_IWIKI_LLM_KEY; не перетирает заранее заданный; no-op при незаданном источнике.
SC4_config_key_allowed принимает ICODEX_IWIKI_LLM_KEY и по-прежнему отвергает raw IWIKI_LLM_KEY.
