Секции intent

Цель
Отчёт должен объяснять, что будет построено и почему это можно одобрить без чтения всех markdown-артефактов.
Желаемые результаты
Вкладки интента, спеки и плана дают описание, схемы, карты покрытия и зависимостей; пользователь принимает именно HTML-отчёт.
Метрики здоровья
Детерминированность проверок, offline HTML, tab-merge contract, совместимость с chain-gate.py и читаемость сохраняются.
Ограничения
HTML-текст только на русском; markdown-артефакты только на английском; внешние ресурсы и CDN запрещены.
Детали для принятия
  • Граница доверия: check-chain остаётся владельцем смыслового payload, потому что он читает исходные артефакты и frontmatter.
  • Граница рендера: html-report остаётся offline renderer/merge layer и не читает chain sources напрямую.
  • JavaScript: малый inline JS допустим; крупная vendored-библиотека требует отдельного обоснования; CDN запрещён.
  • Язык: пользовательский HTML генерируется по-русски, markdown-документы остаются на английском.
  • Review loop: пользователь смотрит HTML-отчёт; замечания исправляются в markdown-источниках; затем отчёт регенерируется.

Обязательная схема: поток принятия

Обязательные semantic views

Карта покрытия требований
Outcome интента связывается с требованием спеки и acceptance-критерием. Это показывает, что в спеках нет лишнего и нет пропущенного.
Граф компонентов
check-chain/SKILL.md владеет смысловым payload; html-report и chain-report.md владеют рендерингом и merge contract.
Поток данных
Markdown artifact → validation/frontmatter → enriched tab payload → offline HTML report → пользовательское принятие.
Карта рисков
Не ломать chain-gate.py, frontmatter, TODO schema, offline-контракт, tab markers и языковое правило.

Карта требований

Цель интентаТребование спекиКритерий принятия
Отчёт объясняет изменения без чтения markdownExecutive overview, approval lens, source anchors, expandable evidenceПользователь может принять stage по HTML-отчёту.
Максимально использовать схемыMandatory rich diagrams для интента, спеки, плана и результатаКаждая проверенная вкладка имеет обязательную схему или fallback matrix.
HTML по-русски, markdown по-английскиLanguage boundary в architecture и validation checksВсе видимые подписи отчёта русские; markdown source остаётся English.
Безопасный review loopHTML-first review workflowЗамечания исправляются в markdown, затем check-chain обновляет отчёт.
Сохранить gate contractsContract compatibility и non-goalschain-gate.py, frontmatter, TODO schema и verdict semantics не меняются.
Границы и риски
  • Не переносим смысл в renderer: html-report не читает markdown-источники в chain-mode.
  • Не меняем gate truth: HTML остаётся пользовательским артефактом, но machine gate читает только frontmatter.
  • Не добавляем тяжёлую библиотеку: первый scope — CSS, inline SVG, <details>, малый inline JS как enhancement.
  • Fallback: если данных для полноценной схемы мало, вкладка показывает компактную матрицу и явную заметку.
